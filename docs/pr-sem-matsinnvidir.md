# Pull requests sem matsinnviðir (PR-arkitektúr)

**Hugmyndin:** PR eru ekki bara tæknilegt atriði – þau eru **pedagógísk skjöl** sem gera samvinnu *
*sýnilega, rekjanlega og metanlega**.

## Hvers vegna PR?

- **Rökstuðningur + breytingasaga** í sama þræði
- **Jafningjarýni** sem krafist er áður en samþykkt er
- **Svar við athugasemdum** er skráð (engin „þögul“ lagfæring)
- **Handover** á sér stað í PR – skýrt hvað var gert og af hverju
- **Atvinnulífslegur ávinningur:** vinnulag sem speglar raunveruleg teymi

## Lágmarksreglur í námsmati

1. Fókus PR (afmarkað umfang + skýr lýsing)
2. Að minnsta kosti **2 efnislegar jafningjarýnir** áður en merge er heimilað
3. **Skyldu-svör** við athugasemdum (ekki „offline“ samkomulag)
4. Merge aðeins eftir samþykki, ólokin atriði fara í **Issue**
5. Uppfæra **README/running‑leiðbeiningar** ef framkvæmd breytist
6. Sýnilegt framlag: höfundur, meðhöfundur, rýnir, eftirfylgni

## Kennslufræðilegur ávinningur

- Byggir upp **fagmennsku, rekjanleika og ábyrgð**
- Minnkar **fríríðslu** með sýnilegu framlagi
- Gagnlegt til **réttlátara einstaklingsmats** í teymum

> Þessi hluti byggir á upplifun og hönnun PR‑miðaðs mats í tveimur tengdum námskeiðum (IE → BI), með
> einni **viðvarandi geymslu** í BI þar sem endurgjöf flyst áfram á milli lota (carry‑forward).
